# Terraform開発ガイドライン（ベーステンプレート）

このファイルは `docs/terraform-guidelines.md` を作成する際のベースです。
プロジェクト開始時にこの内容をベースとし、「プロジェクト固有設定」セクションの内容を埋めて `docs/terraform-guidelines.md` を作成してください。

---

## ディレクトリ構成

```
infra/
├── envs/
│   ├── dev/
│   │   ├── main.tf          # プロバイダー設定・モジュール呼び出し
│   │   ├── variables.tf     # 変数定義
│   │   ├── outputs.tf       # 出力値定義
│   │   └── terraform.tfvars # 変数の実値
│   ├── staging/
│   └── prod/
└── modules/
    ├── {役割・機能名}/
    └── ...
```

- `envs/` : 環境ごとのルートモジュール。環境差分（インスタンス数、CIDR等）はここで管理する
- `modules/` : 再利用可能なモジュール。1モジュール1責務で構成する
- モジュール名はクラウドのサービス名ではなく、**役割・機能名**（`network`, `database`, `kubernetes` 等）を使う

---

## モジュール設計方針

### 原則

- **1モジュール1責務**: モジュールごとに管理するリソースの範囲を明確にする
- **モジュール間の依存は outputs/variables で明示する**: モジュールをまたぐリソース参照は直接参照せず、呼び出し元（`envs/`）を経由して受け渡す

### モジュール内のファイル構成

```
modules/{モジュール名}/
├── main.tf        # リソース定義
├── variables.tf   # 入力変数
├── outputs.tf     # 出力値
└── locals.tf      # ローカル変数（必要な場合のみ）
```

### 公式モジュールの活用方針

IAMロール・セキュリティグループ・アドオンなど、多数のサブリソースを伴う複雑なクラウドサービスには **Terraform公式モジュール（terraform-aws-modules 等）の利用を検討する**。

- 公式モジュールはベストプラクティスに沿った設定を内包しており、自作時にハマりやすい問題を回避できる
- シンプルな構成のリソースは自作モジュールで十分
- 公式モジュールを使用する場合は `~>` でバージョンを固定する
- どのサービスに公式モジュールを使うかはプロジェクト固有設定に記載する

## 命名規則

### リソース命名（クラウド上の `name` 引数）

以下のパターンを標準とする。

```
{システム名}-{環境}-{リソース種別}[-{連番/補足}]
```

- システム名・環境名の具体値はプロジェクト固有設定で定義する
- 一貫性を保つため、全リソースで同じパターンを使用する

### Terraformリソースラベル（`.tf` ファイル内の識別子）

- スネークケースを使用する
- 単一リソースの場合は `this` または `main` など役割を表す名前を使う
- 複数リソースの場合は区別できる名前を付ける

### 変数命名

- 変数名はスネークケースを使用する
- bool型の変数は `enable_` / `is_` プレフィックスを付ける
- リスト型の変数は複数形にする

## バージョン固定

各環境の `main.tf` に `required_version` と `required_providers` を記載し、バージョンを固定する。

- `~>` 演算子でマイナーバージョンまでの変更を許容する（パッチは自動更新、メジャーは手動対応）
- バージョンアップ時は `terraform init -upgrade` を実行し、`.terraform.lock.hcl` をコミットする
- 具体的なバージョンはプロジェクト固有設定に記載する

---

## 変数・アウトプット規約

### 変数（variables）

- 全変数に `description` と `type` を記載する
- `type` は明示する（`any` は原則使用しない）
- デフォルト値はモジュールでは設けてよいが、環境ルートモジュールでは `terraform.tfvars` で指定する
- `validation` ブロックで入力値の制約を表現する（CIDR形式、文字数制限等）

### アウトプット（outputs）

- 他モジュールや外部から参照される値のみ出力する
- 全アウトプットに `description` を記載する
- `sensitive = true` でシークレットのマスクを行う

---

## stateバックエンド

- リモートバックエンド（S3 / GCS / Azure Blob）を使用する。`terraform.tfstate` をローカルに持たない
- state locking を有効にする（DynamoDB / GCS / Azure Blob のネイティブロック）
- stateファイルの暗号化を有効にする
- バックエンドリソース自体は Terraform管理対象インフラとは別に事前準備する
- 具体的な設定値はプロジェクト固有設定に記載する

## git管理対象

| 対象 | git管理 | 理由 |
|---|---|---|
| `.terraform.lock.hcl` | **含める** | プロバイダーバージョンをチーム・CI間で固定するため |
| `.terraform/` | **除外** | プロバイダーキャッシュ（環境ごとに再取得） |
| `tfplan` / `*.tfplan` | **除外** | 一時的なplanファイル |
| `terraform.tfvars` | **原則除外** | 環境変数の実値を含むため（シークレットなしなら管理可） |
| `*.auto.tfvars` | **除外** | 意図しない変数上書きを防ぐため |
| `*.tfstate` / `*.tfstate.*` | **除外** | リモートバックエンド使用時の安全策 |

## Terraform操作フロー

### 実装中の品質チェック（随時実施）

コードを書きながら随時実行する。まとめて最後に実行しない。

```bash
terraform init       # プロバイダー・モジュール追加後に実行
terraform fmt        # フォーマット整形（保存のたびに）
terraform validate   # 構文チェック（モジュール追加・変更後）
```

### apply前の確認（必須）

```bash
terraform plan -out=tfplan    # 差分を確認してユーザーに提示・承認を得る
```

planの確認ポイント:
- 意図しないリソースの削除（`-`）がないか
- 意図しないリソースの再作成（`-/+`）がないか（特に破壊的再作成が危険なリソース）
- 変更（`~`）の内容が想定通りか

### apply / destroy の実行

- `terraform apply tfplan` : 承認済みのplanファイルを適用する（承認後の差分変動を防ぐ）
- `terraform destroy` : 削除対象リソースの一覧を提示し、**必ず承認を得てから実行する**

### 環境の適用順序

```
dev → staging → prod
```

必ず下位環境で動作確認してから上位環境へ適用する。

## セキュリティルール

### 基本方針

- **シークレットをコードに含めない** - APIキー・パスワード・証明書等はSecret Manager / Key Vault等で管理し、tfvarsやstateにも平文で書かない
- **最小権限の原則** - IAMロール・サービスアカウントは必要最小限の権限のみ付与する
- **通信の暗号化** - クラウドリソース間・クライアントとの通信はTLSを使用する
- **セキュリティグループ / ファイアウォールの原則** - デフォルト拒否、必要な通信のみ明示的に許可する

### シークレットの参照方法

シークレットはクラウドのシークレット管理サービスに格納し、Terraformからは `data` ソースで参照する。

```hcl
# 例: 変数にハードコードせず、data ソースで取得する
data "xxx_secret" "example" {
  name = "/project/env/secret-name"
}
```

## リソース検証の方法

`terraform apply` 完了後、**必ず各クラウドのCLI（AWS CLI / gcloud / az）でリソースが正しく作成されたことを確認する**。
Terraformの成功はAPIレスポンスの正常終了を意味するに過ぎず、リソースが正しく動作しているかは別途検証が必要。

特にKubernetesクラスターと通信セキュリティ設定は、構成の複雑さと影響範囲の大きさから、一般的なリソース検証に加えて追加の検証を行う。

### 検証の役割分担

| 担当 | 確認方法 |
|---|---|
| Claude | CLIコマンドを実行して確認・結果を報告 |
| ユーザー | クラウドコンソールで目視確認（任意） |

### 検証のタイミング

- **モジュール単位で apply → 確認 → 次のモジュールへ進む**
- 確認結果（リソースIDなど）は `.steering/*/tasklist.md` に記録する
- 確認でエラーや想定外の状態が見つかった場合は、次のリソース構築に進まず原因を特定・解決する

### 通信セキュリティの検証

セキュリティグループ・ファイアウォール・ネットワークポリシー設定後は、リソースの存在確認だけでなく、**設定通りに通信が許可・拒否されていることを実際に確認する**。

- 許可設定: 対象ポート・プロトコルでの接続が成功すること
- 拒否設定: 対象ポート・プロトコルでの接続がタイムアウト・拒否されること

### Kubernetesクラスターの検証

EKS / GKE / AKS などのクラスター構築後は、以下の両方で確認する。

| ツール | 確認内容 |
|---|---|
| クラウドCLI | クラスター状態・ノードグループ・アクセス設定・IAM設定 |
| `kubectl` | ノード状態（Ready）・システムPodの稼働・クラスターへの疎通 |
